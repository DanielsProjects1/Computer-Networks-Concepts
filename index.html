Networking Concepts

Here are the main networking topics I researched for this project.

DNS (Domain Name System)

DNS is like the internet's phonebook. It converts domain names like github.com into IP addresses that computers can use. The process goes: browser cache → OS cache → recursive resolver → root server → TLD server → authoritative server.

IP Addressing

IPv4 uses 32-bit addresses like 185.199.108.153 and supports about 4.3 billion addresses. IPv6 uses 128-bit addresses like 2606:50c0:8000::153 and has way more. GitHub Pages supports both (dual-stack).

HTTP vs HTTPS

HTTP (port 80) sends data in plain text so anyone can read it. HTTPS (port 443) uses TLS encryption to keep data private. Our site uses HTTPS so it's secure.

Network Security

Security features include TLS certificates, HSTS headers (forces HTTPS), DNSSEC (prevents fake DNS), and firewalls. GitHub Pages automatically gives you HTTPS with a valid certificate.

HTTP Headers

Headers are extra info sent with each request/response. Examples: Content-Type, Cache-Control, Strict-Transport-Security. You can see them using curl -I in the terminal.

Protocols — HTTP vs HTTPS

HTTP (Port 80)

  • Data is sent in plain text
  • Anyone on the network can read it
  • No way to verify the server is real
  • Vulnerable to man-in-the-middle attacks
  • No integrity checks

HTTPS (Port 443)

  • Uses TLS 1.3 encryption
  • Certificate proves the server is legit
  • HSTS header forces HTTPS always
  • Protects against eavesdropping
  • Data integrity is verified

TLS Handshake (simplified)

# How HTTPS sets up a secure connection (TLS 1.3)
 
Client → ClientHello (says what TLS versions/ciphers it supports)
Server → ServerHello + sends its Certificate
Client → Checks certificate against trusted CA
Both → Exchange keys using ECDHE algorithm
Both → Send "Finished" — encrypted connection is ready!
 
Done in ~1 round trip (TLS 1.3 is fast!)
Cipher used: TLS_AES_128_GCM_SHA256
